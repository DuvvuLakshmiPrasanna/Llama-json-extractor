# Prompting vs. Fine-Tuning

Prompt engineering and fine-tuning both improved structured extraction quality, but they solved different failure classes. On the three worst baseline samples, stronger prompt constraints improved parse success from 0/3 to 2/3. This confirms that many base-model failures are instruction-following failures: markdown fences, prose preambles, and partial schema drift can often be reduced by explicit formatting constraints and few-shot templates. Prompting is therefore effective when you need rapid iteration, zero training cost, and can tolerate some response variability.

However, prompt-only control remained fragile. One document still failed repeatedly due to a trailing-comma formatting error even after stricter prompts and few-shot examples. This is a key production insight: prompt edits influence behavior probabilistically, while downstream parsers require deterministic syntax. Across the full 20-document evaluation, fine-tuning improved parse success from 40.0% to 90.0%, raised average key accuracy from 0.696 to 0.989, and improved average value accuracy from 0.546 to 0.881. The largest gain was not semantic extraction alone; it was output regularity under schema constraints.

Fine-tuning also changed the default response prior. After LoRA SFT on curated JSON outputs, the model produced fewer prose wrappers, fewer wrong key names, and tighter type consistency. Remaining failures were narrower: occasional trailing punctuation, missing nested arrays, and numeric string type drift. These residual errors are best addressed with targeted data augmentation rather than additional prompt wording.

In production, use prompt engineering first for low-volume workflows, prototyping, and schema experimentation. Move to fine-tuning when parse reliability is a hard SLA, when failures are systematic across document variants, or when maintaining long prompts becomes operationally expensive. The practical pattern is sequential: prototype with prompts, then lock reliability with high-quality curated SFT data.
